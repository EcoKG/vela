# 수정 전략 — 에러 타입별 접근법

## 컴파일/타입 에러
1. 에러 메시지의 파일:라인 확인
2. 타입 정의와 실제 사용 비교
3. plan.md의 Class Specification에 정의된 타입으로 수정
4. 누락 import 추가

## 런타임 에러
1. 스택 트레이스의 최초 발생 지점 확인
2. null/undefined 체크 누락 확인
3. 비동기 처리 (await 누락, Promise 체인) 확인
4. 경계 조건 (배열 범위, 빈 문자열) 확인

## 테스트 실패
1. 기대값과 실제값 비교
2. plan.md의 Test Strategy와 대조
3. 테스트 자체가 올바른지 확인 (테스트 버그 가능)
4. 모킹이 올바르게 설정되었는지 확인

## 통합 에러
1. API 엔드포인트 URL, 요청/응답 형식 확인
2. DB 스키마와 코드의 컬럼/타입 일치 확인
3. 환경 설정 (환경변수, 포트, 경로) 확인

## 수정 원칙
- **최소 수정**: 1~3줄 변경이 이상적
- **원인 해결**: 증상이 아닌 근본 원인 수정
- **회귀 방지**: 수정 후 전체 테스트 통과 확인
- **최대 3회**: 3회 수정 시도 후에도 실패하면 사용자에게 보고
