# 진단 절차 — 단계를 절대 건너뛰지 않는다

## 1단계: 에러 분석
- 에러 메시지를 정확히 읽는다
- 스택 트레이스에서 **최초 발생 지점**을 식별한다
- 에러 타입을 분류한다:
  - **컴파일/타입 에러** → 타입 불일치, 누락 import
  - **런타임 에러** → null 참조, 범위 초과, 비동기 실패
  - **테스트 실패** → 기대값 불일치, 타이밍, 모킹 문제
  - **통합 에러** → API 불일치, DB 스키마 불일치

## 2단계: 경쟁가설 수립
에러 원인에 대해 **2~3개 가설**을 수립한다:
```
H1: import 경로가 잘못되었을 수 있음
H2: 함수 시그니처가 {N}-plan.xml와 다를 수 있음
H3: 비동기 처리가 누락되었을 수 있음
```

## 3단계: 증거 수집
각 가설에 대해 코드를 읽어 증거를 수집한다:
- 에러 발생 파일 읽기
- 관련 파일 읽기 (import, 의존성)
- {N}-plan.xml의 Class Specification과 대조

## 4단계: 근본 원인 확정
증거에 기반하여 가설을 제거하고 근본 원인을 확정한다.

## 5단계: 수정 적용
- **최소 수정**: 근본 원인만 수정한다
- {N}-plan.xml의 명세를 벗어나지 않는다
- 수정 후 **반드시** 테스트 실행

## 6단계: 결과 보고
아티팩트 디렉토리에 `debug-report.md` 작성:
```markdown
## 에러
{에러 메시지}

## 근본 원인
{확정된 원인}

## 수정 내용
{변경 파일 + 변경 내용}

## 검증
{테스트 실행 결과}
```
