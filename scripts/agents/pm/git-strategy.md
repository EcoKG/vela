# Git 전략 — 태스크 단위 원자적 커밋

## 기본 원칙

**태스크 완료 즉시 커밋한다.** 파이프라인 단위 일괄 커밋 금지.
각 커밋은 하나의 완결된 태스크를 나타내야 한다.

---

## 커밋 메시지 규칙

### 태스크 커밋 (execute 단계)
```
feat({phase}-{task}): {태스크 설명}
fix({phase}-{task}): {태스크 설명}
refactor({phase}-{task}): {태스크 설명}
docs({phase}-{task}): {태스크 설명}
```

예시:
```
feat(1-1): implement JWT middleware
feat(1-2): add user model
fix(2-3): resolve token expiry edge case
```

### Summary 커밋 (execute 단계 완료 후)
```
docs({phase}): add execution summary
```

### Ship 커밋 (ship 단계)
```
chore({phase}): ship — PR #{pr_number}
```

### 마일스톤 태그
```
chore: release v{major}.{minor}.{patch}
```
태그: `v1.0.0`, `v1.1.0`, `v2.0.0` ...

---

## Executor 커밋 절차

태스크 완료 후 즉시:

```bash
# 1. 변경된 소스 파일만 스테이징 (artifacts 제외)
git add {변경된_소스_파일들}

# 2. 태스크 커밋
git commit -m "feat({phase}-{task}): {설명}"

# 3. summary.md 작성 후 커밋
git add .vela/artifacts/{N}-{M}-summary.md
git commit -m "docs({phase}): add {N}-{M}-summary"
```

**summary.md는 반드시 git에 커밋한다** — 다음 세션 Claude가 이전 작업 컨텍스트를 복원하는 데 사용.

---

## Ship (PR 생성) 규칙

`/vela:ship N` 실행 시:

1. `.vela/artifacts/{N}-context.md` — 작업 설명 (PR 제목 소재)
2. `.vela/artifacts/{N}-verification.md` — 검증 결과 (PR 본문 소재)
3. `.vela/artifacts/{N}-*-summary.md` — 변경 내역 (PR 본문 소재)

```bash
gh pr create \
  --title "{N}-context.md에서 추출한 작업 설명" \
  --body "$(cat <<'EOF'
## Summary
{context.md 기반 요약}

## Changes
{summary.md들 기반 변경 내역}

## Verification
{verification.md 기반 검증 결과}

🤖 Generated by Vela v4 Pipeline
EOF
)"
```

`--draft` 플래그 전달 시 draft PR로 생성:
```bash
gh pr create --draft ...
```

---

## 마일스톤 완료 (milestone-complete)

1. 아티팩트 아카이브: `.vela/artifacts/{N}-*/` → `.vela/archive/{N}/`
2. 릴리스 커밋:
   ```bash
   git commit -m "chore: release v{version}"
   git tag v{version}
   ```
3. 태그 푸시: `git push origin v{version}`

---

## 금지 사항

- `git push --force` — 절대 금지
- `git reset --hard` — 절대 금지
- `git commit --no-verify` — 절대 금지
- 보호 브랜치(main/master/develop) 직접 커밋 — 금지
- `.vela/` 내부 파일(훅, 설정) 커밋에 포함 — 금지 (artifacts 제외)
