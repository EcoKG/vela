# 모델 선택 전략 — 반드시 적용

모든 에이전트 소환 시 model 파라미터를 **반드시** 지정한다. 생략 금지.

| 작업 유형 | model 파라미터 | 용도 |
|----------|---------------|------|
| 파일 탐색, 검색, 읽기 | `"haiku"` | Glob/Grep/Read 중심 |
| 코드 구현, 수정, 테스트, 리뷰 | `"sonnet"` | 코딩/리뷰 |
| 설계, 디버깅, 리서치 분석 | `"opus"` | 깊은 사고 |

## 역할별 기본 모델

| 역할 | model | 변경 금지 |
|------|-------|----------|
| Researcher | `"opus"` | 경쟁가설 디버깅에 깊은 분석 필수 |
| Planner | `"opus"` | 아키텍처 설계에 깊은 사고 필수 |
| Executor | `"sonnet"` | 코딩 품질 + 비용 효율 |
| Reviewer | `"sonnet"` | 코드 리뷰 |
| Conflict Manager | `"sonnet"` | 충돌 관리 |
| Debugger | `"opus"` | 근본 원인 분석 |
| 탐색 전용 | `"haiku"` | 빠른 파일 탐색 |

## 자동 모델 에스컬레이션 — 실패 시 반드시 적용

에이전트가 실패하면(결과물 없음, 에러, 불완전 산출물) **상위 모델로 자동 재시도**한다.
수동으로 같은 모델을 재시도하는 것은 **금지**한다.

### 에스컬레이션 체인

```
haiku 실패 (2회)  →  sonnet으로 재시도
sonnet 실패 (2회) →  opus로 재시도
opus 실패 (1회)   →  사용자에게 AskUserQuestion으로 보고
```

### 에스컬레이션 규칙

1. **같은 모델로 최대 2회**까지만 재시도한다 (Opus는 1회)
2. 재시도 시 이전 실패의 에러 메시지/원인을 프롬프트에 **반드시** 포함한다
3. 에스컬레이션 시 역할은 유지하고 model만 변경한다
4. Opus까지 실패하면 **즉시 사용자에게 보고** — 자동 재시도 금지

### 에스컬레이션 프롬프트 형식

```
[이전 시도 실패]
모델: {이전 model}
에러: {에러 메시지 또는 실패 원인}
시도 횟수: {n}회

위 실패를 참고하여 다시 시도하세요.
{원래 태스크 프롬프트}
```

### 에스컬레이션 판단 기준

| 실패 유형 | 판단 |
|----------|------|
| 산출물 파일이 생성되지 않음 | 즉시 에스컬레이션 |
| 산출물이 불완전 (필수 섹션 누락) | 1회 더 시도 후 에스컬레이션 |
| 테스트 실패 | Debugger 에이전트 소환 (에스컬레이션 아님) |
| 런타임 에러 (Claude Code 내부) | 즉시 에스컬레이션 |
| 컨텍스트 초과 | 즉시 에스컬레이션 (상위 모델의 더 큰 윈도우) |

### 에스컬레이션 추적

에스컬레이션 발생 시 아티팩트 디렉토리에 `escalation-log.json`을 작성한다:
```json
{
  "step": "execute",
  "role": "executor",
  "attempts": [
    { "model": "sonnet", "result": "fail", "reason": "테스트 3개 실패" },
    { "model": "sonnet", "result": "fail", "reason": "동일 테스트 실패" },
    { "model": "opus", "result": "pass" }
  ]
}
```
