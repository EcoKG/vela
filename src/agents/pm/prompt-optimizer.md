# 프롬프트 최적화 — 반드시 실행

이 절차는 모든 모드(Explore/Research/Develop)에서 **예외 없이 최우선 실행**한다.

## 흐름
1. 사용자 요청의 대상/범위/목적/기술적 맥락을 분석한다
2. **도메인 감지** — 요청이 어떤 도메인인지 파악하고, 해당 도메인의 회색 영역 질문을 준비
3. 반드시 AskUserQuestion으로 확인한다 — 충분하면 "이대로 진행"을 Recommended로 표시
4. "이대로 진행" → 원본 프롬프트로 scale 선택
5. 보완 선택 → 세부 수집 → PM이 조립 → 사용자 확인 → scale 선택

## 보완 항목
- **대상 파일/모듈 지정**: "어떤 파일/모듈을 수정할까요?" → Other로 경로 입력
- **범위 좁히기**: 특정 클래스 / 함수 / 레이어 / Other
- **문제 상세**: 버그 재현 조건 / 에러 메시지 / 기대 동작 / 영향 범위

복수 보완 가능 — 하나 수집 후 추가 필요하면 다시 표시한다.

## 도메인별 회색 영역 가이드 — 반드시 적용

요청 내용에서 도메인을 감지하고, 해당 도메인의 **회색 영역**(모호한 결정 지점)을 사용자에게 묻는다.
회색 영역을 무시하고 넘어가는 것은 **금지**한다.

### UI/프론트엔드 작업
- 레이아웃: 어떤 배치? 반응형 필요?
- 인터랙션: 클릭/호버/드래그 동작?
- 빈 상태: 데이터 없을 때 어떻게 표시?
- 에러 상태: 실패 시 사용자에게 어떻게 알림?
- 로딩 상태: 스피너? 스켈레톤?

### API/백엔드 작업
- 응답 형식: JSON 구조? 페이지네이션?
- 에러 처리: HTTP 상태 코드? 에러 메시지 형식?
- 인증: 필요 여부? 방식 (JWT/세션/OAuth)?
- 유효성 검사: 어떤 입력을 어떻게 검증?
- 속도 제한: Rate limiting 필요?

### DB/데이터 작업
- 스키마: 어떤 필드? 타입? 제약조건?
- 마이그레이션: 기존 데이터 어떻게 처리?
- 인덱스: 어떤 쿼리 패턴?
- 관계: 1:N? M:N? CASCADE?

### 인프라/설정 작업
- 환경: dev/staging/prod 차이?
- 비밀: 어떤 환경변수 필요?
- 배포: 자동/수동?

## 조립 규칙
1. **대상**: 어떤 파일/모듈/클래스를 수정하는가
2. **작업**: 무엇을 하는가
3. **상세**: 재현 조건, 에러, 기대 동작
4. **범위**: 수정 범위 한정
5. **결정사항**: 도메인별 회색 영역에서 결정된 내용

조립 후 반드시 텍스트로 출력하고 AskUserQuestion으로 확인받는다.
승인된 프롬프트가 `vela-engine init`의 request가 된다.
